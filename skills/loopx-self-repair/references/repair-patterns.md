# LoopX Self-Repair Patterns

Use this file as a compact diagnosis table. Add a row whenever a real incident
teaches a reusable control-plane lesson.

| Pattern | Symptoms | Evidence To Read | Likely Root | Durable Repair |
| --- | --- | --- | --- | --- |
| `state_projection_gap` | `quota should-run` says `should_run=true` or `must_attempt=true`, but user and agent open todo counts are both zero while `Next Action` still contains executable work. | quota payload, active state `## Agent Todo`, `## User Todo`, `Next Action`, `refresh-state` warnings. | Active state prose has drifted away from machine todo projection. | Expand executable prose into `- [ ]` agent todos, add refresh-state warning/smoke if projection missed it. |
| `user_todo_projection_gap` | `interaction_contract.user_channel.action_required=true` or open user todos exist, but heartbeat/final message only says "owner gate" or does not name the concrete decision. | quota payload `interaction_contract`, `user_todo_summary`, active state user todos, recent final messages. | User-channel projection or agent final-response rule is too vague. | Fix prompt/projection so required user todo/question is surfaced; only say "具体 user todo 未投影" when action is required but absent. |
| `lark_inbox_priority_projection_gap` | A configured bot has a pending direct question or mention, but quota/status exposes only a static drain command and ordinary monitor/advancement work runs first. | local-private inbox counts, content-free oldest-pending age, `goal_boundary.capabilities.lark_event_inbox`, `work_lane_contract`, source-thread reply/readback receipt. | Inbox availability was modeled as a prompt reminder instead of a schedulable urgency read model and priority lane. | Project content-free pending/question/mention counts plus oldest age; make `reply_due` preempt ordinary work, then require durable effect, exactly one idempotent source-thread reply with readback, and ACK. Never project message bodies, ids, senders, or private paths into quota/status. |
| `lark_collector_profile_binding_gap` | A `configured_chat_all` inbox promises complete chat capture, but only some messages arrive or the collector and reply paths observe different bot apps; a non-mention present in bounded source history never reaches drain. | ignored collector/inbox config shape, generated service argv with values redacted, effective Lark CLI profile identity, inbox counts, bounded source-history presence, and exactly-once drain/ACK receipt. | The host collector invoked `lark-cli event consume` through the machine default profile while the reply path used a registered project profile; chat filtering and urgency classification were correct but applied to the wrong event stream. | Require an explicit non-default collector profile, or reuse the enabled reply sender profile; reject collector/reply mismatch, place `--profile` before every live collector event command, expose only content-free binding provenance, and cover non-mention ingest plus restart-safe dedupe/ACK. |
| `reviewer_reader_auth_misclassified_as_identity_drift` | A reviewer sink has explicit user-reader and bot-sender bindings, the sender bot remains healthy, but an unavailable reader user credential is reported as an identity mismatch and the agent starts rewriting profiles instead of requesting login. | Goal-default sink shape with profile values redacted, content-free `auth status --verify` availability for the configured reader and sender, provider call count, and secondary notification blocker. | Runtime user authentication availability and static profile-binding validity were collapsed into one identity-mismatch blocker. | Preserve the two explicit bindings; return a content-free reader-auth-required blocker, project the existing scoped login action, and perform zero destination reads or sends until user authentication verifies. Cover unavailable reader auth with a focused sink smoke. |
| `normal_no_user_todo_misflagged` | `action_required=false` and user open count is zero, but the agent reports a state projection fault. | quota payload and user todo summary. | Rule over-applied the "todo not projected" fallback. | Allow "无用户待办/无需通知"; update heartbeat/prompt smoke if needed. |
| `runtime_capability_user_gate_overreach` | A missing launcher/runtime capability such as `network` becomes `capability_gate.action=ask_owner`, `requires_user_action=true`, or a user notification even though no credential, protected access, or production authority is needed. | quota `capability_gate`, heartbeat `--available-capability` inputs, launcher/runtime observations, interaction contract user channel. | Capability availability and permission authority were collapsed into one owner-held class. | Keep runtime provisioning gaps in the agent repair lane, require truthful runtime observation before declaring availability, and reserve `ask_owner` for credentials or protected/production authority. Cover all-blocked and mixed-capability cases with capability-gate and monitor smokes. |
| `blocked_priority_fallback_hidden` | A core P0 path is user-gated, safe P1/P2 fallback runs, but the user is not told what blocked the P0 path. | quota `safe_bypass_allowed`, active priority stack, final heartbeat text. | Fallback execution hid the higher-priority blocked item. | Report concrete blocker plus fallback progress; expose `blocked_priority_fallback` in state/payload if missing. |
| `boundary_projection_gap` | `recommended_action` asks for work in a scope that is absent from `goal_boundary.write_scope`, such as runner edits blocked by repo-only scope. | quota `recommended_action`, `goal_boundary.write_scope`, authority/owner decision records. | Checkpointed owner decision did not project into current write boundary. | Repair boundary projection or ask controller/user; do not consume repo-only handoff steps that cannot touch the needed scope. |
| `public_surface_internal_context_leak` | Public docs, examples, fixtures, catalogs, README links, PR descriptions, or shipped evidence contain internal planning labels, meeting/reporting context, local-only status narratives, private operator framing, raw coordination details, or references to ignored local artifacts. | `git diff --stat`, `git diff --name-only`, `git grep`/`rg` over tracked public surfaces, docs indexes, fixture schema names, PR refs/history if already pushed, `.gitignore`, `.local` target path. | The agent treated a public artifact as durable memory for private planning, or generalized too late after internal context had already entered tracked history. | Move private planning to `.local` or another ignored state store, rewrite public text into generic product/maintainer language only when it is truly reusable, scan current public heads and PR refs, add a user/support gate if pushed read-only refs or cached views still expose the context, and record a concrete prevention todo before resuming feature work. |
| `periodic_report_editorial_contract_gap` | A periodic-report artifact claims operational context is collapsed, but the first real hosted report puts normalized-document parity, renderer/digest details, delivery receipts, archive-provider canaries, or other process narration in the hero or primary sections; a later readability pass may instead delete primary facts, omit supporting facts from copied text, or leave controls untranslated even when the profile language is set. A hosted shell may also expose supporting metadata or fail direct section hashes after dynamic mount. | Normalized document item kinds and visibility, compiled editorial lineage, primary and supporting item counts, original-to-rendered fact checklist, renderer `content_policy`, Markdown/HTML completeness, localized hosted DOM first viewport, initial hash position, source snapshot refs/digests. | Audience versus operational content was a prose convention instead of a normalized document contract, and readability was treated as deletion rather than information hierarchy; an authored hero summary could bypass item visibility, and language metadata did not own the built-in UI vocabulary. The host may also re-render the document without preserving renderer visibility semantics. | Compile hero summaries only from typed primary outcomes, risks, and next actions, record exact item/field lineage, and make renderers reject authored or stale summaries; preserve audience facts through bounded named detail rows; require runtime and delivery-receipt items to be supporting; collapse them in HTML and retain them in a labeled Markdown appendix; localize built-in controls from the profile language; validate source-to-artifact counts and representative facts; publish HTML artifacts without re-rendering or validate presentation parity; restore section hashes after mount; treat mutable source links as navigation rather than frozen evidence. |
| `shared_adapter_local_path_leak` | A shared adapter projection, such as Lark Kanban sync, would write an absolute local checkout, runtime, active-state, raw-log, or artifact path into a remote/shared row. | adapter dry-run payload, generated record values, docs boundary section, public/private scan. | Adapter reused local executor context as shared control-plane metadata. | Strip or replace the local path at the adapter projection layer, add a focused smoke that rejects the path, and only store explicit worker launch context when the row was authored for that purpose. |
| `peer_workspace_boundary_gap` | A peer edits or prepares repository changes from a non-git directory, unrelated worktree, or checkout that violates the selected task/repository isolation policy; a cross-repository repair may also be misclassified against the goal repo. | `agent_identity`, selected todo `task_repository` and write scopes, goal `repo`, current git origin/worktree/root, quota payload, heartbeat prompt commands, AGENTS worktree policy. | Workspace safety was inferred from identity labels, prompt text, or the goal repo alone instead of the selected task and `quota should-run`. | Add or repair `agent_workspace_guard_v1`; use a credential-free first-class `task_repository` only for genuine cross-repository tasks, and require a matching linked worktree. Block delivery and spend until the peer moves to a compliant worktree and reruns the guard with the same `--agent-id`. Keep the goal repo fallback and do not treat the field as scope or write authority. Read-only and monitor-only work should not be blocked merely by identity. |
| `implicit_review_route_drift` | `todo complete` invents a default reviewer, routes completion by an agent profile, or blocks the completing peer without an explicit review request. | failing `todo complete` preview, successor continuation policy, todo claim metadata, generated heartbeat prompt, todo lifecycle smoke. | Review ownership survived as an identity-level parent/default handoff instead of task policy. | Use typed continuation: ordinary `independent_handoff` is non-blocking, `same_agent_non_delivery` stays with the peer, and only explicit `review_handoff` to a different registered peer creates a blocking dependency. Profiles must not supply an implicit reviewer. |
| `nonblocking_pr_review_continuation_gap` | A validated feature PR is waiting for human review and the same agent has independent runnable work, but the source todo remains blocked or the generated user todo becomes `user_gate`, so quota stops normal delivery. | source completion command, generated user/agent todos, user todo `task_class`/binding/gate scope, PR monitor, quota interaction contract, branch base topology. | `--next-user-todo` defaulted to a blocking gate and the workflow treated review latency as merge authority; no explicit reminder-versus-gate choice was recorded. | Complete the feature slice, derive a bound `user_action` with `--next-user-task-class user_action`, keep a runnable agent successor, and add a separate `continuous_monitor` for PR lifecycle readback. Use a stable integration branch when stacking experimental feature PRs and reserve `user_gate` for the exact aggregate merge/release/launch or protected authority boundary. Cover the generated todo metadata and eligible quota path in the todo lifecycle smoke. |
| `nonblocking_review_empty_frontier_replan_gap` | A bound `user_action` remains open after a feature PR, no runnable agent successor exists, and two bounded stalls are visible, but status still reports `operator_gate` or the replan only records prose. | active-state user/agent todo summaries, `waiting_on`, run-history stall evidence, replan obligation `todo_actions`, execution/interaction contracts, refreshed repair delta. | Active-state routing required an existing agent todo before treating `user_action` as non-blocking, and empty-frontier replans did not require concrete todo writeback. | Keep pure `user_action` as a visible non-blocking notice even on an empty agent frontier. After the stall threshold, require `autonomous_replan_required` to create and claim a runnable agent todo and record `runnable_todo_set`; allow terminal no-follow-up only with closure evidence. Preserve `user_gate` precedence for exact authority boundaries. |
| `global_route_collision` | A peer runs `connect`/`bootstrap` with an existing `goal_id` from a different repo or state file, and the dashboard appears to lose state even though the original files still exist. | global registry entry, `source_registry`, `repo`, `state_file`, project-local registry, active state path, recent `sync-global` or `connect` payload. | `sync-global` merged by `goal_id` and silently rewrote the global route pointer from a non-authoritative checkout. | Fail closed on route changes unless `--replace-state` is explicit and backed up; use `register-agent` through the existing `source_registry` when adding a peer, or choose a new `--fork-goal` id for a true fork. Cover with a focused connect-route-collision smoke. |
| `guided_goal_identity_reuse` | `start-goal --guided` runs in a project with multiple registered goals and silently reuses the first registry goal or its only registered agent, even though the request asks for a new peer/meta lane. | guided start packet, project-local goal order/status, requested versus selected `goal_id`/`agent_id`, host-loop activation identity gate, shared global registry after `register-agent`. | Guided startup treated registry order and single-agent convenience selection as user intent; agent registration could also sync through a project-local runtime instead of the shared registry it originally read. | Return a machine-readable goal-selection gate until `--goal-id` is explicit; when the user asks for a new peer, register a distinct public-safe `--agent-id` before todo writeback and host-loop activation. Keep `register-agent` synchronization pinned to the same shared global registry route and cover both seams with focused smokes. |
| `global_registry_write_boundary_gap` | `connect`, `sync-global`, or agent registration can write/read project-local state, but writing the shared global registry fails with permission or read-only host errors; the user sees an ambiguous "not connected" or a half-updated control plane. | command payload `global_sync`, `global_registry_writability`, project-local registry, global registry path, `loopx doctor`, filesystem write probe, recent source registry contents. | The shared global registry is a required projection for multi-project status/quota, but command flow treated it as best-effort or discovered write denial after mutating the source registry. | Probe global registry writability before source/local mutations when global sync is required; fail fast with `ok=false`, `global_registry_write_denied`, and recovery guidance. Keep explicit local-only mode behind `--no-global-sync`, and prefer `register-agent` through `source_registry` for agent onboarding. |
| `runtime_projection_source_mirror_ambiguity` | A project-local material refresh appends its source run, but shared-runtime projection reports `ambiguous` when the source runtime's registry mirror and exactly one external runtime both declare the same source registry. | refresh-state route diagnostics, source and candidate runtime roots, each registry's `source_registry`, compact route match counts, and an immediate shared-runtime readback. | Route discovery counted the source runtime's local registry mirror as an independent projection target alongside the authoritative external runtime. | Prefer external matches when any exist, report the ignored source-mirror count separately, and preserve ambiguity when multiple external targets remain. Cover source mirror plus one external target and source mirror plus two external targets, then verify the real refresh route readback. |
| `agent_scoped_user_gate_overreach` | `quota should-run --agent-id <agent-a>` returns `user_gate`, `requires_user_action=true`, or `delivery_allowed=false` because an open user todo exists, but that todo is scoped to `<agent-b>` through `blocks_agent` or `claimed_by`; agent A has an independent runnable todo or only needs its own workspace/scope repair. | quota payload with the same `--agent-id`, `user_todo_summary.first_open_items`, `blocks_agent`, `claimed_by`, `agent_lane_next_action`, active state user/agent todos, related interaction pattern. | User-gate projection treated agent-scoped routing metadata as diagnostic text instead of a hard scope boundary, so a target-agent gate became a global owner gate. | Filter other-agent `blocks_agent` / `claimed_by` user todos out of the current agent's blocking summary while preserving diagnostic visibility; if the raw state was `operator_gate` only because of that other-agent gate and the current agent has runnable work, project an eligible current-agent lane. Cover with `examples/control_plane/quota-agent-scoped-user-gate-smoke.py` and workspace-guard smoke coverage. |
| `user_action_binding_projection_gap` | A non-blocking user action intended for one peer appears in another peer's heartbeat, or its scoped summary has `open_count=0` while inherited goal state still produces `operator_gate` / `NOTIFY`. | Active user-todo metadata, registered agents, agent-scoped quota summary and diagnostics, inherited status state, interaction contract user channel, todo mutation actor. | User actions had no first-class goal/agent binding, `claimed_by` was overloaded as routing, and quota filtered item details without clearing the upstream operator-gate state. | Require every multi-agent user todo to declare `bound_agent` or `goal_bound`; keep gate scope separate, make the bound agent the response-continuation lifecycle actor, filter other-lane details and counts, and override stale operator-gate state before interaction projection. Preserve `claimed_by` only as a read compatibility fallback and cover the no-current-work quiet lane. |
| `same_agent_scoped_gate_due_monitor_gap` | A same-agent user gate has an exact `unblocks_todo_id` or narrow `decision_scope`, an unrelated capability-runnable continuous monitor is due, but quota returns `must_attempt=false` and blocks the whole agent lane. | quota payload `scoped_user_gate_fallback`, `agent_todo_summary.monitor_due_items`, capability gate, gate relation, linked todo, and current-agent monitor metadata. | Scoped-gate fallback considered only advancement candidates, so the separately projected due-monitor lane disappeared before dependency filtering. | Add the already scoped, due, capability-runnable monitor projection to scoped-gate dependency filtering and preserve normal todo priority ordering; keep linked work blocked and reject monitors with missing capabilities. Cover both positive and negative cases in `examples/control_plane/quota-agent-scoped-user-gate-smoke.py`. |
| `mixed_monitor_capability_projection_split` | One due monitor is capability-runnable and another is capability-blocked, but quota selects the runnable monitor while `capability_gate.action=skip`, the work-lane due count includes both, and the interaction contract claims every candidate is unavailable. | quota `selected_todo`, `work_lane_contract.monitor_due_items`, `agent_todo_summary.monitor_due_count`, capability-gate runnable/blocked candidates, and the same `--available-capability` inputs. | Capability filtering split due monitors into runnable and blocked projections, then the capability gate consumed only the blocked projection while the work-lane count recomputed from unfiltered monitor items. | Feed both runnable and blocked due-monitor projections into the capability gate, keep blocked diagnostics, and make projected runnable due count authoritative for the work lane. Cover a mixed private-read/network pair in `examples/control_plane/monitor-scheduler-contract-smoke.py`. |
| `monitor_debt_advancement_starvation` | Overdue monitor catch-up produces repeated unchanged no-spend turns while same- or higher-priority advancement remains runnable; a scoped user-gate fallback may keep reselecting the monitor even after the ordinary work lane chose advancement. | current-agent run history excluding scheduler/accounting/state-refresh bookkeeping, monitor material-change flags, scoped fallback selection, work-lane reason codes, and Lark inbox urgency. | Monitor fairness was applied only to the ordinary work-lane contract; scoped fallback rebuilt and re-sorted candidates without the recent unchanged-monitor streak. | After two consecutive unchanged monitor-only work turns, prefer same- or higher-priority runnable advancement in every candidate-selection seam and expose compact arbitration evidence. Any non-monitor work or material monitor transition resets the streak; `lark_event_inbox.reply_due` still preempts all ordinary lanes. Cover scoped fallback starvation, terminal transition reset, and reply-due precedence. |
| `unscoped_multi_agent_user_gate` | A peer sees a `user_gate` todo that belongs to another lane because the todo has no `blocks_agent` and no explicit `global_gate=true`; every `quota should-run --agent-id ...` call may project it as a current owner gate. | active-state user todo metadata, registered agents, quota `user_todo_summary`, current `--agent-id`, CLI authoring command, interaction contract user channel. | The authoring layer allowed open multi-agent user gates without machine scope, relying on prose and agent memory to infer which lane was blocked. | Reject new multi-agent `user_gate` writes unless they specify `blocks_agent` / `--agent-id` or explicit `global_gate=true` / `--global-gate`; make `loopx check` flag existing unscoped gates so old/manual state must be repaired. Cover with `examples/control_plane/todo-user-gate-scope-smoke.py`. |
| `consumed_user_gate_writeback_gap` | The user says a projected review or action already happened, but quota keeps returning `user_gate` / `operator_gate` and no successor or replan appears. | exact user todo id and lifecycle metadata, latest user statement or typed external evidence, quota before and after lifecycle writeback, current agent vision and runnable frontier. | The completion fact stayed in chat or an external surface instead of transitioning the durable todo, so gate precedence correctly hid the empty-frontier replan. | Complete only when an explicit decision outcome grants or rejects the exact scope; otherwise supersede the stale gate without consuming authority. Refresh state, rerun scoped quota, and require a successor or `autonomous_replan_required` when an active vision still has no runnable advancement. |
| `agent_scoped_no_candidate_gap` | `quota should-run --agent-id <peer>` returns `should_run=true`, `delivery_allowed=true`, and `quiet_noop_allowed=false`, but `agent_lane_next_action` is empty, `current_agent_claimed_advancement_items` is empty, and no runnable candidate is projected for that peer; the recommendation may still point at another peer's lane. | quota payload with the same `--agent-id`, active state claimed todos, registered peer profile/scope, recent history, open review blockers. | Agent-scoped quota does not distinguish "goal has runnable work" from "this peer has runnable work"; other-agent or out-of-scope candidates keep the heartbeat eligible. | Project `reassignment_required` when only other-peer claims remain, or `agent_scope_wait` for an explicit blocking review dependency. Allow quiet no-op/no-spend until a concrete current-agent or unclaimed todo is available. |
| `agent_scoped_replan_broadcast_gap` | `quota should-run --agent-id <peer>` returns `autonomous_replan_required` from an unscoped replan obligation even though that peer has no runnable advancement work and should wait quietly or run only due monitor work. | quota payload with the same `--agent-id`, `autonomous_replan_scope`, `goal_route_hint`, obligation owner fields/triggers, active agent/monitor todo summaries. | Goal-level replan was treated as a broadcast obligation before peer task ownership was resolved. | Apply explicit replans only to their owner; resolve unscoped replans through the current goal frontier or deterministic task coordination, and let unrelated peers fall back to `reassignment_required`, `agent_scope_wait`, or `monitor_quiet_skip`. Cover with the replan decision-plane smoke. |
| `agent_scoped_spend_attribution_gap` | A peer writes a validated accountable delivery and then runs scoped `quota spend-slot`, but `delivery_run_generated_at`, classification, and agent id are null after another peer writes a newer refresh or spend event. | scoped spend payload, requested `agent_id`, recent goal run index ordered by time, same-agent accountable refresh, intervening other-agent refresh/spend events. | Latest-unspent lookup stopped at the first goal-wide event instead of scanning the requested agent's event stream, so concurrent peers broke causal attribution. | For explicit `agent_id`, skip events attributed to other agents and stop only on same-agent or legacy unscoped terminal/accounting events; keep unscoped lookup behavior unchanged. Cover target delivery -> other-agent delivery/spend -> target scoped spend with identity assertions. |
| `agent_scoped_cli_identity_projection_gap` | `quota should-run --agent-id <peer>` identifies the registered peer, but generated `refresh-state`, `monitor-poll`, or `spend-slot` actions omit `--agent-id`; copied commands then append goal-scoped runs or accounting rows with `agent_id=null`. | quota `agent_identity`, interaction-contract `next_cli_actions`, recent run index agent ids, and scoped versus unscoped command fixtures. | Interaction-command rendering treated peer identity as diagnostic context instead of a causal writeback/accounting argument on every generated state command. | When the source quota call has an agent id, preserve it on every generated refresh, monitor, spend, and follow-up quota command; keep deliberately unscoped/controller calls unchanged and cover both scoped and unscoped command packets. |
| `deferred_gate_resume_misclassified` | A ready deferred successor with `resume_when` or `unblocks_todo_id` is visible, but quota/status treats the peer turn as no-candidate, `agent_scope_wait`, or quiet no-op. | quota payload deferred lanes, active state deferred todo metadata, `resume_condition`, `resume_ready`, claimed_by split, related user/review gate todo. | Deferred work was modeled as absence of todo instead of a gate-resume lifecycle condition. | Route ready current-agent/unclaimed deferred work through `successor_replan_required` / `deferred_resume_projection`; keep deferred visibility after open lanes; only fall through to no-candidate wait when no ready deferred resume candidate exists. |
| `resume_gated_open_todo_selected` | An open todo with `resume_when` and `resume_ready=false` appears in `first_executable_items`, `executable_backlog_items`, `capability_gate.runnable_candidates`, `agent_lane_next_action`, or `recommended_action`; a lower-priority safe fallback exists but is hidden behind the not-yet-unlocked successor. | quota payload with the same `--agent-id`, active-state todo metadata for the selected todo and its `resume_when` target, `resume_condition.target_status`, status `agent_todo_summary`, catalog canary plan/run output. | The shared executable-candidate path treated visible open todos as runnable after one projection seam already filtered them elsewhere. | Make open todos with `resume_when` non-executable until `resume_ready=true` across all executable projections, add a focused quota smoke for the positive and negative cases, and wire the touched policy seam into catalog canary domain checks before generic family checks. |
| `pr_merge_event_projection_gap` | Issue-fix lifecycle state records a terminal merged PR and its monitor closes, but a successor with `resume_when=pr_merged:#N` remains `resume_ready=false`. | issue-fix PR lifecycle row, completed monitor evidence, rollout event log, successor `resume_condition`. | The lifecycle writeback persisted domain state without appending the public `pr_merge` rollout event consumed by todo resume projection. | When a connected lifecycle writeback observes `MERGED`, append one repo-qualified, public-safe, idempotent `pr_merge` event; replay must reuse the same event and immediately satisfy the dependent resume condition. |
| `pr_merge_resume_repository_collision` | A todo with `resume_when=pr_merged:#N` becomes ready because an unrelated repository merged the same PR number, or a cross-repository dependency cannot be distinguished from its execution repository. | todo `task_repository`, qualified/unqualified resume ref, compact public `pr_merge` rollout events, projected `resume_condition`. | Resume matching treated a bare PR number as globally unique and ignored the todo's repository identity. | Bind unqualified refs only to the todo's GitHub `task_repository`; require `owner/repo#N` for cross-repository dependencies; fail closed with a machine-readable ambiguity when no repository can be derived. Cover identical PR numbers across repositories without reading bodies or comments. |
| `handoff_gate_state_projection_gap` | A `blocks_agent` handoff/review todo changes status, but the blocked agent's quota gate does not change immediately: open review work may be hidden behind stale done handoffs, or a done review without a successor falls through to generic `agent_scope_wait`. | quota payload `agent_scope_frontier`, status `agent_todos.handoff_gates`, active-state todo metadata `status`, `done`, `blocks_agent`, `unblocks_todo_id`, `superseded_by`, and successor links. | Handoff lifecycle was inferred from open todo lanes instead of a small state machine, so done/superseded handoff todos vanished from gate inputs or stale done gates outranked live blockers. | Project `todo_handoff_gate_v0` from the complete todo list: `blocking`, `cleared_without_successor`, `cleared_with_successor`, `superseded`, and `deferred`. Quota should prefer current-agent `blocking` gates, map `cleared_without_successor` to `successor_replan_required`, run concrete successors normally, and ignore superseded stale gates. Cover with `examples/control_plane/quota-cleared-blocker-successor-gate-smoke.py`. |
| `inter_agent_handoff_priority_gap` | A peer has same-priority runnable work, including explicit review todos that unblock another peer via `blocks_agent` and `unblocks_todo_id`, but the handoff is buried behind ordinary backlog. | quota `agent_lane_next_action`, `capability_gate.runnable_candidates`, todo continuation metadata, registered peer identities, active state Next Action. | Explicit review dependencies were visible but absent from the runnable-candidate tie-breaker. | Rank current-peer claimed unblock/review work ahead of ordinary same-priority backlog in both selected-action surfaces; add a smoke that the selected lane and first runnable candidate agree. |
| `task_coordinator_worker_drift` | A peer selected as temporary coordinator repeatedly executes one child lane itself while other eligible child lanes remain open under `multi_subagent`. | quota `task_orchestration_contract`, goal orchestration policy, claimed-by-others lanes, work-lane contract, `interaction_contract.agent_channel.primary_action`, generated heartbeat prompt. | Child lanes were diagnostic-only and the task-scoped coordinator contract still said to execute the first todo directly. | Project eligible child lanes into `task_orchestration_contract`; tell the selected coordinator to spawn/resume the bounded bundle and aggregate accepted evidence. Do not persist coordinator status as agent rank. |
| `stale_recommended_action` | The agent repeats "launch or preflight" after the active state or evidence already narrowed the next action. | latest active state, quota payload timestamp/content, recent history rows. | Recommended action was not regenerated from the newest state. | Refresh state, update recommender/projection tests, and write an exact next action. |
| `stale_active_next_priority_override` | A lower-priority todo named by durable `Next Action` becomes `agent_lane_next_action` even though the same peer has a higher-priority capability-runnable todo, commonly after a deferred P0 resumes or another peer refreshes shared state. | scoped quota payload, `capability_gate.runnable_candidates`, `agent_lane_next_action`, active-state `Next Action`, todo priority/resume/claim metadata, recent peer writes. | The active-next hint was ranked before todo priority, so stale steering text acted as a global override instead of a same-priority tie-breaker. | Preserve peer claim/profile routing, then rank todo priority before active-next preference; keep active-next as a same-priority tie-breaker. Cover a resume-ready P0 plus lower-priority active-next and concurrent other-peer work. |

| `benchmark_launcher_sandbox_mode_drift` | A host-Codex benchmark exits before case materialization with a sandbox-probe failure even though the same runner and credential worked in a prior batch. | launcher dry-run, requested/effective host Codex sandbox mode, preflight category, runner source fingerprint. | The public launcher did not forward the sandbox mode used by the validated runner profile, so the runner fell back to an unsupported host sandbox probe. | Expose and forward the host Codex sandbox setting without weakening its default; make dedicated runners opt into any broader mode explicitly and record only the mode label in public prerequisites. |
| `user_reward_lesson_projection_gap` | The user explicitly corrects a high-value product route, priority, or operating rule, but later turns follow an older todo/recommended_action as if the correction never happened. | recent user correction, latest quota/status, active-state `Next Action`, open todos, recent run history, interaction pattern catalog. | The correction stayed in chat/model belief instead of becoming a durable human-reward/operating-lesson constraint, successor todo, or state projection. | Promote the correction into active goal state and pattern catalog; add/update an agent todo for runtime projection; refresh state so `quota should-run` selects the corrected rule before continuing. |
| `reward_feedback_leak` | Later benchmark rounds receive verifier tail, pass/fail, or reward details that one arm should not see. | benchmark prompt/continuation logs, compact ledger, verifier wrapper. | Evaluation loop leaked reward signal. | Record per-round reward privately for metrics, but do not feed it to agents unless the experiment explicitly studies feedback. |
| `commit_hygiene_drift` | Broad commit includes temporary smokes, raw logs, local state, or unrelated docs. | `git status`, `git diff --stat`, `git ls-files --others --exclude-standard`, AGENTS.md. | Worktree was staged by chronology rather than reviewer logic. | Use explicit pathspecs, split commits, keep only durable smokes, and update AGENTS/skill if the failure mode recurs. |
| `docs_surface_sprawl` | Root docs become hard for contributors to navigate; research/drafts/history compete with stable contracts. | `docs/README.md`, root docs count, docs governance smoke. | Documentation lacks audience and lifecycle ownership. | Move archive/outreach/research/reference material into indexed subdirs and keep new docs linked from an index. |
